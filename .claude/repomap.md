# Repo Map

_Last updated: 2026-05-27. Generated by /repomap — edit `.claude/repomap-config.json` to change what is included._

## `backend/app/`

Root Flask application package. Contains the full backend layer stack: api (routes), db (engines/sessions), domain (ORM models), repositories, services, schema (Pydantic), plus middleware, logging, openapi helpers, gateway, core, and utils. Entry point is __init__.py which wires up the app factory.

## `backend/app/domain/`

Domain rule modules — state-aware business logic that sits between services and repositories. One file per domain area: access, auth, permissions, public links, submissions, surveys, users, versions. Also contains errors.py defining the canonical error classes (409 conflict, 422 invalid payload, 500 server bug) used across the stack.

## `backend/app/repositories/`

Named query helpers — one file per domain entity (projects, surveys, users, submissions, roles, members, access, invitations, public links, content, response stores). Each repo targets a single database with no cross-db coordination. The only layer permitted to run queries directly against the ORM models.

## `backend/app/services/`

The only layer permitted to orchestrate across both databases. Contains business logic and saga workflows for all major domains: surveys, projects, submissions, users, members, roles, access, public links, public surveys, results, content, and audit logging. Cross-db operations spanning core + response DBs are coordinated here exclusively.

## `backend/app/schema/`

Pydantic and ORM schema layer, split into api/ and orm/. api/ contains request/response models (with enums, limits, and per-domain request/response subdirs); orm/ contains SQLAlchemy declarative models split by database — core/ (CoreBase) and response/ (ResponseBase), which must never be mixed. Never import ORM models into Pydantic schemas.

## `backend/app/api/v1/`

Versioned HTTP route handlers. Top-level: auth, health, me, public. Project-scoped routes under projects/ cover surveys, content, members, roles, submissions, versions, and public links. All handlers are thin — parse the request, call a service, return a response. No SQL or business logic permitted here.

## `backend/tests/`

Test suite split into unit/ and integration/. Unit tests cover services, auth, schemas, and logging. Integration tests run inside Docker via run-tests.sh and cover core DB, response DB, environment, and model-layer behaviour. All sessions use savepoint-based fixtures — commits release savepoints, outer transaction rolls back on teardown.

## `frontend/apps/studio-app/src/api/`

All API interaction for the Studio app. client.ts provides the base useApi() hook; types.ts holds shared response types; generated/ contains auto-generated OpenAPI TypeScript types (do not edit). Domain hooks are in auth/, me/, project/, survey/ subdirs. Never call fetch directly — always use these hooks with TanStack Query.

## `frontend/apps/studio-app/src/routes/`

TanStack Router file-based routes for the Studio SPA. Top level: index (dashboard), account. Project routes under projects/$slug/ cover members, roles, settings, surveys. Survey routes under $surveySlug/ cover builder, access, links, members, roles, responses, overview, settings, and versions. Add pages by creating files here — never edit routeTree.gen.ts by hand.

## `frontend/apps/studio-app/src/components/`

Studio-specific React components not tied to a single route. Layout pieces (SiteHeader, StudioSidebar, Breadcrumb), access control UI (SurveyAccess, ProjectPermissionGate, PermissionBadge), role/member helpers (MemberRoleActions, RoleBadgePreview), forms (CreateProjectForm, CreateSurveyForm), and auth/ subdir for auth-related components.

## `frontend/apps/public-site/src/pages/`

Astro file-based pages for the public site. Currently: index.astro (landing page) and docs/[...slug].astro (dynamic docs route). Static .astro components ship zero JS — React islands with client:* directives are used only where interactivity is needed.

## `frontend/packages/`

Shared libraries consumed by both studio-app and public-site. Four packages: ui (@flowform/ui — generic component library), styles (@flowform/styles — design tokens and Tailwind v4 config), site-shell (@flowform/site-shell — SiteHeader and nav constants), builder (@flowform/builder — survey/question builder). Packages must never import from app-level code.

## `tools/mcp/`

Two FastMCP servers sharing a single uv venv (pyproject.toml). flowform_dev.py exposes OpenAPI discovery tools (list_endpoints, find_endpoints, describe_endpoint, describe_schema) fed from a live backend. repomap.py orchestrates iterative repo summarisation, writing one scoped rule file per path into .claude/rules/repomap/. Each server has its own run script.
