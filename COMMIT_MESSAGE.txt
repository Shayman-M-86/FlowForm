feat: implement real respondent session start; consolidate enums and DB-CHECK typing

Replaces the Phase-2 public session-start placeholder with a working
SessionStarter-backed implementation, introduces the subject-resolution
services and core session repositories, consolidates all Literal enums into a
single shared schema module, and types every DB-CHECK-backed ORM column against
those enums. Also removes legacy submission ORM/domain code, adds a small
domain guard, and rounds out test coverage for the new session logic.

Full suite: 290 passing.

--------------------------------------------------------------------------------
Public session start: stub -> real implementation
--------------------------------------------------------------------------------

- app/api/v1/public.py: the public "start submission session" route no longer
  returns a placeholder. It now parses the request, resolves the optional
  authenticated user, calls SessionStarter().start(...), and sets the real
  browser-session cookie (set_submission_session_cookie) with the issued raw
  token. The route is now @auth.optional_auth() so an authenticated respondent
  can be recognised while anonymous access still works.
- app/api/v1/public_submission_session_temp.py: updated alongside the contract.

New service layer under app/services/submissions/:

- session_starter.py — SessionStarter orchestrates a session start: resolve
  access, ensure a response store, resolve the optional subject, create the core
  session, conditionally consume single-use links, commit, and build the public
  response.
- access_resolver.py — SurveyAccessResolver resolves either public-slug or
  link-token access to a SubmissionAccessGrant (survey + published version +
  optional link), raising the appropriate domain errors on each failure path.
- project_subject_resolver.py — ProjectSubjectResolver derives the optional
  stable subject from server-owned context only (assigned link, authenticated
  identity, or recognition token); browser-supplied IDs are never trusted.

New core repositories under app/repositories/core/:

- submission_sessions.py (session creation + browser-token hashing)
- project_subjects.py, project_subject_identities.py, project_subject_tokens.py

--------------------------------------------------------------------------------
Shared enums: one source of truth
--------------------------------------------------------------------------------

- Moved all Literal enum aliases out of app/schema/api/enums.py into a new
  app/schema/enums.py that sits below both the ORM and API schema sub-layers, so
  each can import the aliases without crossing sideways (ORM must not import
  api/, api/ must not import orm/). Repointed ~20 importers.
- Removed inline / duplicated Literals across the API request and response
  schemas (e.g. SurveyResponseSessionStatus, *AnswerState, export format) in
  favour of the shared aliases (SubmissionSessionStatus, SubmissionAnswerState,
  ExportFormat, etc.).

--------------------------------------------------------------------------------
ORM columns typed against their DB CHECK constraints
--------------------------------------------------------------------------------

Every enum-style column previously typed Mapped[str] now carries its shared
Literal alias, mirroring the CHECK constraints in the core SQL schema (columns
stay TEXT — no native Postgres ENUM, no migration):

- surveys.visibility, survey_versions.status
- project_memberships.status, project_invitations.status
- response_stores.store_type
- project_subject_identities.identity_type / verification_status
- submission_sessions.session_status, submission_events.event_type

This surfaced one latent bug: invitations_repo.update_status took a bare str and
assigned it to the now-typed column; its parameter (and the parallel
members_repo.update_membership) were narrowed to the enum alias, pushing the
guarantee to the call boundary.

--------------------------------------------------------------------------------
Domain layer cleanup
--------------------------------------------------------------------------------

- Added app/domain/guards.py with ensure_present(value, *, error), a single
  Optional-narrowing guard now used across ~13 service/resolver call sites in
  place of ad-hoc None checks.
- Added app/domain/submission_access_rules.py holding ensure_link_token_access
  (a real five-check composite); the slug-access pass-through wrapper was
  inlined at its one call site and removed.
- Deleted dead domain modules: submission_rules.py, user_rules.py.
- Added SubjectResolutionError (500, server-invariant) raised when a
  link/identity/token references a subject that cannot be resolved — a defensive
  guard against a broken FK invariant rather than a silent downgrade to an
  anonymous session.
- SubmissionAccessGrant moved into services/results.py (service-layer result
  DTO) instead of living in the domain layer.

--------------------------------------------------------------------------------
Legacy submission storage removed
--------------------------------------------------------------------------------

Deleted the old submission ORM models and their result wrapper now that the
session/response model supersedes them:

- schema/orm/core/survey_submission.py, response_subject_mapping.py
- schema/orm/response/submission.py, submission_answer.py, submission_event.py
- results.LinkedSubmissionResult and the matching entries in the ORM registry
  (schema/orm/__init__.py, schema/orm/core/__init__.py)
