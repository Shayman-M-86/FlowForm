# Identity and authentication

Defines how an Auth0 identity becomes an authenticated FlowForm user. Project
and survey authorization is owned by [[projects-and-access|Projects and access]];
respondent recognition that does not require login is owned by
[[links-and-subjects|Links and subjects]].

## Purpose

This domain gives operator and authenticated-respondent requests one externally
verified identity and one local user record. Auth0 remains the credential and
login provider, while FlowForm keeps the minimum local account state needed for
authorization, invitations, account display, and links to project subjects.

## Responsibilities

- Validate required and optional bearer tokens against the configured Auth0
  issuer and API audience. Optional authentication permits an absent token but
  rejects a malformed or invalid one.
- Bootstrap a local user from an Auth0 ID token after proving that its subject
  matches the already-verified access-token subject.
- Persist the Auth0 subject, normalized email, display name, public identifier,
  durable email-verification flag, and platform-administrator flag in core data.
- Create a default project, owner role, project membership, and platform response
  store when a user is first bootstrapped.
- Coordinate profile, email, username, password-ticket, MFA-reset, verification,
  and account-deletion operations with the Auth0 Management API.

## Non-responsibilities

- It does not decide what a user may do inside a project or survey.
- It does not treat a respondent recognition cookie or survey-link token as an
  Auth0 login.
- It does not store Auth0 passwords, MFA secrets, bearer tokens, ID tokens, or
  Management API access tokens in the application databases.
- It does not own response encryption or the separation between core and
  response data; see [[responses-and-encryption|Responses and encryption]].

## Main entities and invariants

| Concept | Current representation | Invariant |
| --- | --- | --- |
| Auth0 identity | `users.auth0_user_id` | Unique locally and taken from a verified token subject. |
| Local user | `users` | Email is normalized but intentionally not globally unique; external subject, not email, is login identity. |
| Public user identifier | `users.public_id` | Eight URL-safe characters generated by PostgreSQL and unique. |
| Email verification | `users.email_verified` | Durable local flag used by invitation acceptance; it is not continuously synchronized from every ID-token claim. |
| Platform administrator | `users.platform_admin` | Bypasses the project/survey permission decorators; membership-filtered list/read paths do not all apply that bypass. The schema permits the flag only on user ID 1. |

The bootstrap flow verifies an RS256 ID token for issuer, client audience,
expiry, and issued-at presence, then requires its `sub` to equal the access-token
`sub`. Existing users are resolved directly by that subject and are not
re-created from each incoming token.

## Important workflows

1. A client obtains Auth0 access and ID tokens and calls the authenticated
   bootstrap endpoint.
2. The backend validates the access token at the route boundary, validates and
   cross-checks the ID token for a new account, then commits the local user and
   default project state.
3. Later protected routes resolve `users.auth0_user_id` from the verified bearer
   subject and pass the local actor to RBAC or respondent-access policy.
4. Account mutations call Auth0 first where external identity state is involved,
   then persist the mirrored local fields when required.
5. Email verification can be established by a live Management API check or by
   successful possession of a project-invitation token.

## Implementation map

- `backend/app/middleware/auth/auth0.py` owns bearer extraction, access-token
  validation, ID-token verification, and the Management API client.
- `backend/app/services/auth.py` and `backend/app/services/users.py` own local
  bootstrap and default-project creation.
- `backend/app/services/account.py` coordinates Auth0 and local account changes.
- `backend/app/api/v1/account/` exposes bootstrap, profile, verification,
  invitation, and account-lifecycle endpoints.
- `backend/app/schema/orm/core/user.py` and
  `infra/database/init/schema/flowform_core_db_schema_v4.sql` define persisted
  account invariants. Unit tests cover representative required/optional auth and
  bootstrap behavior.

## Verified gaps and open questions

- Changing an email resets verification in Auth0, but the current local update
  does not reset `users.email_verified` or evict its verification cache entry.
- Sensitive account operations rely on an ordinary valid bearer token; no
  recent-login or step-up-authentication policy is implemented in the backend.
- OIDC metadata and JWKS data are cached in-process without an observed
  unknown-key refresh path. Rotation and cache-expiry behavior need an explicit
  operational contract and tests.
- The repository does not establish how `platform_admin` is granted, audited,
  reviewed, or revoked outside the database constraint.
- Auth0-first account changes and deletion are not transactional with the core
  database. Recovery ownership for a later database failure remains undefined.
