# Answer Flows

First saves, changed answers, clearing answers, canonical-answer rules, and save idempotency.

## 17. Flow: saving the first answer for a question

### 17.1 Browser request

The frontend sends:

```text
question_node_id
answer value
client_mutation_id
```

`client_mutation_id` is a random UUID generated by the browser for one intentional save operation.

If a network retry occurs, the browser reuses the same mutation ID.

### 17.2 Validate the session

The backend:

1. hashes the resume token;
2. loads the core session;
3. locks the session row for mutation;
4. confirms the status is `in_progress`;
5. confirms the session has not expired;
6. updates `last_activity_at`.

The save and completion paths should both lock the same core session row so a completion cannot race with a final answer save.

### 17.3 Validate the answer

Load the frozen compiled survey schema.

Confirm:

* the question-node UUID exists;
* the node is answerable;
* the answer family matches;
* the answer value passes the node constraints;
* the question is valid within the current rule path where required.

Never trust frontend validation alone.

### 17.4 Load the response envelope and DEK

Derive:

```text
session_locator
```

Load the response envelope.

Get the plaintext DEK:

* from the in-memory DEK cache; or
* by calling KMS `Decrypt`.

### 17.5 Derive the answer locator

Calculate:

```text
answer_locator
```

from:

```text
session UUID
question-node UUID
linkage secret version
```

### 17.6 Create the first logical answer and revision

Within one response-database transaction:

1. Generate `answer_id`.
2. Generate `revision_id`.
3. Set `revision_number = 1`.
4. Generate a fresh nonce.
5. Serialize the encrypted payload.
6. Build the AAD.
7. Encrypt locally with AES-256-GCM.
8. Insert the stable logical-answer row.
9. Insert the immutable revision row.
10. Point `latest_revision_id` to the first revision.
11. Commit.

The logical answer is now canonical.

### 17.7 Insert analytics metadata

After the response transaction commits, insert a core event:

```text
answer_saved
```

The encrypted response write is authoritative.

If the analytics insert fails, do not tell the respondent that the answer was lost. Log the analytics failure for repair.

## 18. Flow: saving a changed answer

When the participant changes an answer, do not overwrite the previous revision.

### 18.1 Locate the logical answer

Derive:

```text
answer_locator
```

Find:

```text
response_answers
```

by:

```text
envelope_id
answer_locator
```

### 18.2 Lock the logical-answer row

Use a response-database row lock:

```sql
SELECT ...
FROM response_answers
WHERE envelope_id = :envelope_id
  AND answer_locator = :answer_locator
FOR UPDATE;
```

This serialises simultaneous saves for the same question.

### 18.3 Insert a new revision

Read the current latest revision number.

Calculate:

```text
new_revision_number = current_revision_number + 1
```

Generate:

```text
new revision UUID
fresh 12-byte nonce
new ciphertext
```

Insert the immutable revision row.

### 18.4 Move the canonical pointer

Update:

```text
response_answers.latest_revision_id
```

to the new revision.

Commit the response transaction.

The earlier ciphertext remains untouched.

## 19. Flow: clearing an answer

Clearing an answer is a meaningful state change.

Do not delete:

```text
response_answers
```

Do not delete earlier revisions.

Insert a new encrypted revision:

```json
{
  "payload_version": 1,
  "question_node_id": "uuid",
  "state": "cleared",
  "answer": null
}
```

Move the latest pointer forward.

This allows the backend to distinguish:

```text
question was never answered
question was answered and later cleared
```

## 20. Canonical answer rule

FlowForm does not currently need a separate confirmation pointer.

Use this rule:

```text
The latest successfully committed revision is the canonical answer.
```

When a session is completed:

```text
The canonical latest revisions become the final submitted answers.
```

After completion:

```text
Reject further respondent edits by default.
```

A separate `confirmed_revision_id` is only necessary if the product later introduces an explicit draft-versus-confirmed workflow.

## 21. Save idempotency

### 21.1 Why idempotency is needed

A response-database transaction may commit successfully while the HTTP response is lost.

The browser may retry the request.

Without idempotency, the retry creates a duplicate historical revision.

### 21.2 Mutation ID

The browser generates:

```text
client_mutation_id = random UUID
```

for each intentional save.

Reuse the same ID on retries.

Store the mutation ID with the encrypted revision.

### 21.3 Recommended small schema addition

Add:

```text
response_answer_revisions.client_mutation_id UUID NOT NULL
```

Enforce:

```text
UNIQUE (answer_id, client_mutation_id)
```

When a duplicate mutation ID is received:

* return the already-created revision;
* do not insert another revision;
* treat the save as successful.

### 21.4 First-save race handling

Two requests may attempt to create the first logical answer simultaneously.

The existing unique constraint on:

```text
(envelope_id, answer_locator)
```

prevents two logical-answer rows.

Handle the collision by:

1. attempting the insert;
2. catching the unique-constraint conflict;
3. reloading the existing logical answer;
4. continuing through the normal changed-answer path;
5. respecting `client_mutation_id`.
