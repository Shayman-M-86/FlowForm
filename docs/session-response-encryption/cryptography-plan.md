# Cryptography Design Plan (Not Yet Implemented)

> **NOT IMPLEMENTED.** This is a forward-looking design document. None of the
> mechanisms described below  -  KMS envelope encryption, linkage-secret HMAC
> locators, AES-256-GCM answer encryption, or plaintext-key caching  -  exist
> in the codebase today.
>
> The `response_envelopes` and `response_answers` tables already contain
> placeholder columns for this design (Phase 1 schema work): `session_locator`,
> `wrapped_dek`, `kms_key_arn`, `nonce`, `crypto_version`, `kms_context_version`,
> `answer_locator`, `linkage_key_version`, etc. These columns currently hold no
> real cryptographic material  -  they are reserved slots awaiting the
> implementation described here.
>
> A repository-wide search confirms this:
>
> ```text
> grep -rniE "KMS|GenerateDataKey|wrapped_dek|AES" backend/app/services/ backend/app/domain/
> -> zero matches
> ```
>
> No KMS client, no `GenerateDataKey` call, no `AESGCM` usage, and no
> `wrapped_dek` handling exist anywhere in `services/` or `domain/`. This
> document defines what should be built, not what has been built.

## 1. Identifiers and secrets overview

Several distinct values participate in the submission flow. They must not be
confused with one another.

### 1.1 Core submission-session UUID

`submission_sessions.id`  -  a random UUID generated by the backend,
representing one attempt to complete one survey version. Stored only in the
core database. It is the private input used to derive the response-side
session locator.

### 1.2 Browser resume token

A high-entropy (>=32 random bytes) token that lets a respondent return to an
in-progress session. The raw token is sent to the browser; only its SHA-256
hash (`browser_session_token_hash`) is stored in the core database. A
standard SHA-256 is sufficient because the token has high entropy and is not
a human password.

Transport: an `HttpOnly`, `Secure`, `SameSite=Lax` cookie (e.g.
`flowform_submission_session`), distinct from admin auth cookies and from
public survey link tokens. `SameSite=None` for embedded surveys needs a
separate cookie-policy review.

### 1.3 Access-link token

Allows a respondent to begin a survey via a restricted or general survey
link. The core database stores only its prefix and hash. Used during access
resolution; not the same as the resume token.

### 1.4 Linkage secret

Lets the backend derive hidden locators connecting a core session to
anonymous response records. **Decision:** never stored in either Postgres
database  -  held as a random, versioned secret in AWS Secrets Manager,
protected at rest by a dedicated KMS key. [AWS-3]

Example naming: `flowform/prod/submission-linkage/v1`, `.../v2`. The active
version is configured via `FF_LINKAGE_ACTIVE_VERSION`. Each submission
session and each response envelope stores its own `linkage_key_version`. Old
versions stay available as long as old sessions need retrieval or deletion.

### 1.5 Response Key Encryption Key (KEK)

A dedicated symmetric KMS key (e.g. `flowform-response-envelope-kek`) used
only to generate and unwrap per-session DEKs  -  never to encrypt answer
values directly.

### 1.6 Response Data Encryption Key (DEK)

Each submission session gets its own random 256-bit DEK, protecting every
answer revision in that session's response envelope. The plaintext DEK
exists only transiently in backend memory; the wrapped (encrypted) copy
lives in `response_envelopes.wrapped_dek`.

## 2. Linkage key / session locator design

### 2.1 Why the linkage secret and the response DEK are handled differently

**Linkage secret**  -  produces deterministic HMAC locators. The backend may
need to compute locators frequently: starting a session, resuming a session,
saving an answer, retrieving answers, deleting responses, verifying
decrypted payloads. **Decision:** retrieve the secret from Secrets Manager,
cache it briefly in server memory, and compute HMACs locally.

**Response DEK**  -  encrypts answer payloads. Each session gets its own DEK.
AWS KMS `GenerateDataKey` returns both a plaintext copy (used immediately,
then discarded) and an encrypted copy (the only form persisted, in
`wrapped_dek`). [AWS-1]

### 2.2 Deferred alternative: KMS HMAC (`GenerateMac`)

AWS KMS supports HMAC keys via `GenerateMac`. [AWS-4] A stricter design could
send every locator input to KMS so the raw linkage key material never leaves
KMS. The trade-off: every locator calculation becomes a remote AWS call,
answer saves depend on KMS availability, latency/cost increase, and local
development gets harder.

**v1 decision:** Secrets Manager secret, protected by a dedicated KMS key,
cached briefly by the backend, used locally for HMAC-SHA-256. The KMS-HMAC
approach remains a future higher-isolation option, not adopted now.

### 2.3 Session locator derivation

```text
session_locator =
    HMAC-SHA-256(
        linkage_secret_vN,
        "flowform:session-locator:v1" ||
        submission_session_id.bytes
    )
```

A 32-byte value, stored only in `response_envelopes.session_locator`.

### 2.4 Answer locator derivation

Identifies one logical question answer within a session:

```text
answer_locator =
    HMAC-SHA-256(
        linkage_secret_vN,
        "flowform:answer-locator:v1" ||
        submission_session_id.bytes ||
        question_node_id.bytes
    )
```

Stored only in `response_answers.answer_locator`.

### 2.5 Purpose-label versioning

Purpose labels prevent the same input from being used ambiguously across
different cryptographic operations: `flowform:session-locator:v1`,
`flowform:answer-locator:v1`. Build locator messages from fixed UTF-8 labels
and UUID byte representations  -  never informal string concatenation. Bump
the trailing version when the label's format changes.

### 2.6 Why the question ID is both encrypted and indirectly represented

The response database needs to upsert one logical answer when a participant
changes a response. The HMAC-derived `answer_locator` enables this lookup
without exposing the real question ID. The encrypted answer payload still
contains the real question-node UUID; after decryption, the backend
recomputes the answer locator and verifies it matches the row  -  giving a
defensive integrity check against row substitution.

## 3. Response DEK/KEK envelope encryption design

### 3.1 Creating the session DEK (`GenerateDataKey`)

When a response envelope is created:

1. Generate the response-envelope UUID in the backend.
2. Build a non-sensitive KMS encryption context.
3. Call AWS KMS `GenerateDataKey`, requesting an AES-256 data key.
4. Receive a plaintext DEK and an encrypted DEK.
5. Store the encrypted DEK (`wrapped_dek`) in the response envelope.
6. Keep the plaintext DEK temporarily in memory.
7. Drop the plaintext DEK reference once no longer needed.

Conceptual boto3:

```python
result = kms_client.generate_data_key(
    KeyId=response_kek_arn,
    KeySpec="AES_256",
    EncryptionContext={
        "application": "flowform",
        "purpose": "response-envelope-dek",
        "context_version": "1",
        "envelope_id": str(envelope_id),
    },
)

plaintext_dek = result["Plaintext"]
wrapped_dek = result["CiphertextBlob"]
```

`GenerateDataKey` is the correct tool for envelope encryption; direct KMS
`Encrypt` is intended for small values and is not the normal path for
application payload encryption. [AWS-1] [AWS-7]

### 3.2 KMS encryption context

Supply an encryption context for both generation and unwrapping, e.g.:

```text
application     = flowform
purpose         = response-envelope-dek
context_version = 1
envelope_id     = response_envelopes.id
```

The context is authenticated but not secret, and the exact same context must
be supplied on decrypt. [AWS-2] **Do not include**: user IDs, session UUIDs,
project IDs, link tokens, resume tokens, or answer values  -  KMS encryption
contexts can appear in CloudTrail logs. [AWS-2]

### 3.3 `kms_context_version` vs `crypto_version`

Add `response_envelopes.kms_context_version` as a separate version counter
from `response_envelopes.crypto_version`:

- `crypto_version`  -  describes the local (AES-256-GCM) answer-encryption
  format.
- `kms_context_version`  -  describes how the KMS encryption context is
  reconstructed for that envelope.

These evolve independently.

### 3.4 Unwrapping an existing DEK

When the plaintext DEK isn't already cached:

1. Load the response envelope.
2. Read `wrapped_dek` and `kms_key_arn`.
3. Rebuild the exact KMS encryption context (per `kms_context_version`).
4. Call KMS `Decrypt`.
5. Cache the returned plaintext DEK briefly in process memory.

Conceptual boto3:

```python
result = kms_client.decrypt(
    CiphertextBlob=envelope.wrapped_dek,
    KeyId=envelope.kms_key_arn,
    EncryptionContext=build_kms_context(envelope),
)

plaintext_dek = result["Plaintext"]
```

For symmetric ciphertext blobs, KMS can identify the key without being given
the key ID, but specifying the expected key ensures the application only
decrypts with the intended key. [AWS-5]

## 4. AAD construction and AES-256-GCM answer encryption

### 4.1 Algorithm choice

Use **AES-256-GCM** via the high-level `AESGCM` class from the Python
`cryptography` package  -  authenticated encryption with associated data,
protecting confidentiality, ciphertext integrity, and metadata integrity. The
authentication tag is included with the ciphertext output. [CRYPTO-1]

### 4.2 Encrypt-everything rule

Encrypt every answer payload consistently  -  selected choice IDs, ratings,
numeric answers, dates, emails, phone numbers, short text, long text, and
cleared-answer state alike. No exceptions for types that look "harmless";
uniform encryption avoids accidentally exposing answers based on their
apparent data type.

### 4.3 Plaintext payload format

A versioned JSON object, serialized before encryption.

Answered example:

```json
{
  "payload_version": 1,
  "question_node_id": "f1d9a5cb-9184-45a5-a2a6-cfdedc497a89",
  "state": "answered",
  "answer": {
    "value": "Example answer"
  }
}
```

Cleared example:

```json
{
  "payload_version": 1,
  "question_node_id": "f1d9a5cb-9184-45a5-a2a6-cfdedc497a89",
  "state": "cleared",
  "answer": null
}
```

Clearing an answer never deletes history  -  it inserts a new encrypted
revision representing the cleared state.

### 4.4 Nonce generation and uniqueness

Generate a fresh 12-byte random nonce per encrypted revision:

```python
nonce = secrets.token_bytes(12)
```

Never reuse a nonce with the same DEK. The response database enforces
`UNIQUE (envelope_id, nonce)`. Because every historical nonce remains stored
in the revision table, this constraint guards against accidental reuse
across the full lifetime of a session's DEK.

### 4.5 AAD fields and helper signature

Build stable Additional Authenticated Data (AAD) per encrypted revision from:

```text
crypto_version
envelope_id
answer_id
answer_locator
revision_id
revision_number
```

Helper shape:

```python
def build_answer_aad(
    *,
    crypto_version: int,
    envelope_id: UUID,
    answer_id: UUID,
    answer_locator: bytes,
    revision_id: UUID,
    revision_number: int,
) -> bytes:
    ...
```

AAD is not encrypted, but its integrity is authenticated  -  preventing a
database attacker from silently swapping encrypted revision rows.

### 4.6 Encrypt operation

```python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM

aesgcm = AESGCM(plaintext_dek)

ciphertext = aesgcm.encrypt(
    nonce,
    plaintext_payload_bytes,
    aad,
)
```

### 4.7 Decrypt operation and tamper-fail conditions

```python
plaintext_payload_bytes = aesgcm.decrypt(
    revision.nonce,
    revision.ciphertext,
    aad,
)
```

Decryption fails if any of the following is incorrect: key, nonce,
ciphertext, authentication tag, or AAD.

## 5. Plaintext-key caching

### 5.1 Rationale

Requesting a plaintext DEK from KMS on every answer save would be wasteful  - 
a respondent may save many answers during one active session. Cache
plaintext DEKs temporarily in backend process memory.

### 5.2 DEK cache design

Bounded in-memory cache keyed by `response_envelope_id`. Each entry holds the
plaintext DEK and an expiry time. Recommended behavior: short TTL, bounded
max size, LRU eviction, no disk/database persistence, no logging. A process
restart simply clears the cache  -  KMS `Decrypt` is called again as needed.

### 5.3 No shared plaintext-key cache initially

Do not place plaintext DEKs in PostgreSQL, Redis, a message queue, local
files, environment variables, or application logs. In a multi-instance
deployment, each backend instance unwraps and caches its own DEK
independently.

### 5.4 Linkage-secret cache

A separate in-memory cache for versioned linkage secrets retrieved from
Secrets Manager. AWS recommends client-side caching of Secrets Manager values
for speed and reduced API calls. [AWS-3] This cache may use a longer TTL than
the DEK cache, since the secret is shared across sessions and only changes
during intentional rotation.

### 5.5 Python zeroization caveats

Python cannot guarantee perfect cryptographic zeroization of immutable byte
objects. Minimize risk anyway: avoid unnecessary copies, keep cache entries
short-lived, drop references after use, never persist plaintext keys, and
never include them in exceptions or logs.

## 6. Key rotation

Different key types rotate differently.

### 6.1 Response KEK rotation

The response-envelope KEK wraps per-session DEKs.

- **New envelopes:** use the currently active response KEK ARN.
- **Existing envelopes:** keep the original `kms_key_arn` and continue
  decrypting wrapped DEKs with that original key. Existing ciphertext does
  not need rewriting immediately  -  a future maintenance process may rewrap
  old DEKs under a newer KEK.

### 6.2 Linkage-secret rotation

The linkage secret affects deterministic locators and must not be replaced
without versioning. Process:

1. Generate a new random linkage secret.
2. Store it as a new Secrets Manager secret version or versioned secret name.
3. Increment the active linkage version.
4. Use the new version for newly created sessions.
5. Retain old versions for old sessions.
6. Continue reading each session via its stored `linkage_key_version`.

Example: session A -> linkage version 1, session B -> linkage version 2. The
backend loads the correct secret per session.

### 6.3 Crypto-format rotation

`response_envelopes.crypto_version` identifies the local encryption format. A
future version may change the algorithm, payload format, AAD layout, nonce
rules, or serializer. Each envelope is decrypted according to its own stored
`crypto_version`.
