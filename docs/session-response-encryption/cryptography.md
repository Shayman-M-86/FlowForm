# Cryptography

Identifiers, secrets, anonymous locators, KMS envelope encryption, answer encryption, and plaintext-key caching.

## 6. Important identifiers and secrets

Several values are involved in the submission process. They must not be confused with each other.

### 6.1 Core submission-session UUID

```text
submission_sessions.id
```

This is a random UUID generated by the backend.

It represents one attempt to complete one survey version.

It is stored only in the core database.

It is the private input used to derive the response-side session locator.

### 6.2 Browser resume token

The browser resume token allows the respondent to return to an in-progress session.

Generate at least 32 random bytes using a cryptographically secure random-number generator.

Example conceptual value:

```text
random 256-bit token
```

Send the raw token to the respondent browser.

Store only a hash in the core database:

```text
browser_session_token_hash = SHA-256(raw_browser_token)
```

Because the raw token is randomly generated with high entropy, a standard SHA-256 hash is sufficient for lookup. This is not a human password.

The preferred transport is a cookie configured as:

```text
HttpOnly
Secure
SameSite=Lax
```

Use an explicit cookie name such as:

```text
flowform_submission_session
```

The cookie protects the anonymous submission session. It is separate from administrator authentication and separate from a public survey link.

For embedded surveys that require cross-site cookies, review the cookie policy separately before enabling `SameSite=None`.

### 6.3 Access-link token

A link token allows a respondent to begin a survey through a restricted or general survey link.

The core database stores only its prefix and hash.

The link token is used during access resolution. It is not the same thing as the resume token.

### 6.4 Linkage secret

The linkage secret allows the backend to derive hidden locators that connect the core session to anonymous response records.

It must never be stored in either PostgreSQL database.

Use a random versioned secret stored in AWS Secrets Manager.

Secrets Manager protects the secret at rest using a separate KMS key. [AWS-3]

Example secret naming:

```text
flowform/prod/submission-linkage/v1
flowform/prod/submission-linkage/v2
```

The active secret version is configured by the backend:

```text
FF_LINKAGE_ACTIVE_VERSION=1
```

Each core submission session stores its `linkage_key_version`.

Each response envelope also stores its `linkage_key_version`.

Old secret versions must remain available for as long as old sessions need to be retrieved or deleted.

### 6.5 Response Key Encryption Key

Create a symmetric KMS encryption key dedicated to wrapping response DEKs.

Example purpose:

```text
flowform-response-envelope-kek
```

This KMS key is not used to directly encrypt every answer value.

It is used to generate and unwrap per-session DEKs.

### 6.6 Response Data Encryption Key

Each submission session receives its own random 256-bit Data Encryption Key, or DEK.

One DEK protects every answer revision belonging to one response envelope.

The plaintext DEK exists temporarily in backend memory.

The encrypted copy is stored in:

```text
response_envelopes.wrapped_dek
```

## 7. Why the linkage key and response DEK are handled differently

The linkage secret and the response DEK serve different purposes.

### 7.1 Linkage secret purpose

The linkage secret creates deterministic HMAC locators.

The backend may need to calculate locators frequently:

* when starting a session;
* when resuming a session;
* when saving an answer;
* when retrieving answers;
* when deleting responses;
* when verifying decrypted payloads.

The linkage secret should therefore be retrieved from Secrets Manager and cached briefly in server memory.

The backend calculates HMAC values locally.

### 7.2 Response DEK purpose

The DEK encrypts answer payloads.

Each submission session receives a separate DEK.

AWS KMS generates the DEK and returns:

* a plaintext copy for immediate local use;
* an encrypted copy for storage.

The backend stores only the wrapped copy in PostgreSQL. [AWS-1]

### 7.3 Alternative linkage-key design

AWS KMS supports HMAC keys and the `GenerateMac` operation. [AWS-4]

A stricter architecture could send every locator input to KMS and ask KMS to calculate the HMAC.

That has one advantage:

```text
The raw linkage key material never leaves KMS.
```

However, it also creates disadvantages:

```text
Every locator calculation becomes a remote AWS request.
Answer saves depend directly on KMS network availability.
Latency and cost increase.
Local development becomes harder.
```

For the first FlowForm implementation, use:

```text
Secrets Manager secret
    protected by a dedicated KMS key
    cached briefly by the backend
    used locally for HMAC-SHA-256
```

Keep the KMS HMAC approach as a future higher-isolation option.

## 8. Deriving anonymous locators

### 8.1 Session locator

The session locator identifies the response envelope.

Use HMAC-SHA-256 with a purpose label:

```text
session_locator =
    HMAC-SHA-256(
        linkage_secret_vN,
        "flowform:session-locator:v1" ||
        submission_session_id.bytes
    )
```

The result is a 32-byte value.

Store it only in:

```text
response_envelopes.session_locator
```

### 8.2 Answer locator

The answer locator identifies one logical question answer within a session.

```text
answer_locator =
    HMAC-SHA-256(
        linkage_secret_vN,
        "flowform:answer-locator:v1" ||
        submission_session_id.bytes ||
        question_node_id.bytes
    )
```

Store it only in:

```text
response_answers.answer_locator
```

### 8.3 Purpose labels

Purpose labels prevent the same input from being used ambiguously across different cryptographic operations.

Keep each format versioned:

```text
flowform:session-locator:v1
flowform:answer-locator:v1
```

Do not build locator messages from informal string concatenation.

Use fixed UTF-8 labels and UUID byte representations.

### 8.4 Why the question ID is encrypted and also indirectly represented

The response database needs to upsert one logical answer when the participant changes a response.

The HMAC-derived answer locator allows this lookup without exposing the real question ID.

The encrypted answer payload should still contain the real question-node UUID. After decryption, the backend recomputes the answer locator and verifies that it matches the row.

This gives the backend a defensive integrity check.

## 9. KMS envelope encryption

### 9.1 Creating the session DEK

When a response envelope is created:

1. Generate the response-envelope UUID in the backend.
2. Build a non-sensitive KMS encryption context.
3. Call AWS KMS `GenerateDataKey`.
4. Request an AES-256 data key.
5. Receive a plaintext DEK and an encrypted DEK.
6. Store the encrypted DEK in the response envelope.
7. Keep the plaintext DEK temporarily in memory.
8. Remove the plaintext DEK reference when it is no longer needed.

Conceptual Boto3 call:

```python
result = kms_client.generate_data_key(
    KeyId=response_kek_arn,
    KeySpec="AES_256",
    EncryptionContext={
        "application": "flowform",
        "purpose": "response-envelope-dek",
        "context_version": "1",
        "envelope_id": str(envelope_id),
    },
)

plaintext_dek = result["Plaintext"]
wrapped_dek = result["CiphertextBlob"]
```

KMS `GenerateDataKey` is the correct tool because it is designed for envelope encryption. Direct KMS encryption is intended for small values and is not the normal path for application payload encryption. [AWS-1] [AWS-7]

### 9.2 KMS encryption context

Use a KMS encryption context when generating and unwrapping the DEK.

Example:

```text
application     = flowform
purpose         = response-envelope-dek
context_version = 1
envelope_id     = response_envelopes.id
```

The encryption context is authenticated but not secret.

The same exact context must be supplied during decryption. [AWS-2]

Do not include:

* user IDs;
* subject IDs;
* session UUIDs;
* project IDs;
* link tokens;
* resume tokens;
* answer values.

KMS encryption contexts can appear in CloudTrail logs. [AWS-2]

### 9.3 Recommended small schema addition

Add a separate context-format version:

```text
response_envelopes.kms_context_version
```

This is independent from:

```text
response_envelopes.crypto_version
```

`crypto_version` describes local answer encryption.

`kms_context_version` describes how the KMS encryption context is reconstructed.

### 9.4 Unwrapping an existing DEK

When the plaintext DEK is not already cached:

1. Load the response envelope.
2. Read `wrapped_dek`.
3. Read `kms_key_arn`.
4. Rebuild the exact KMS encryption context.
5. Call KMS `Decrypt`.
6. Cache the returned plaintext DEK briefly in process memory.

Conceptual Boto3 call:

```python
result = kms_client.decrypt(
    CiphertextBlob=envelope.wrapped_dek,
    KeyId=envelope.kms_key_arn,
    EncryptionContext=build_kms_context(envelope),
)

plaintext_dek = result["Plaintext"]
```

For symmetric ciphertext blobs, KMS can identify the key without being given the key ID. However, specifying the expected KMS key is recommended so the application only decrypts with the intended key. [AWS-5]

## 10. Local answer encryption

### 10.1 Encryption algorithm

Use:

```text
AES-256-GCM
```

Use the high-level `AESGCM` class from the Python `cryptography` package.

AES-GCM is authenticated encryption with associated data.

It protects:

* confidentiality;
* ciphertext integrity;
* additional authenticated metadata integrity.

The output includes the authentication tag with the ciphertext. [CRYPTO-1]

### 10.2 Encrypt all answer values

Encrypt every answer payload consistently, not only long-text fields.

This includes:

* selected choice IDs;
* ratings;
* numeric answers;
* dates;
* email addresses;
* phone numbers;
* short text;
* long text;
* cleared-answer state.

Uniform encryption avoids accidentally exposing sensitive answers simply because their data type appears harmless.

### 10.3 Plaintext payload format

Serialize a versioned JSON object before encryption.

Example:

```json
{
  "payload_version": 1,
  "question_node_id": "f1d9a5cb-9184-45a5-a2a6-cfdedc497a89",
  "state": "answered",
  "answer": {
    "value": "Example answer"
  }
}
```

For a cleared answer:

```json
{
  "payload_version": 1,
  "question_node_id": "f1d9a5cb-9184-45a5-a2a6-cfdedc497a89",
  "state": "cleared",
  "answer": null
}
```

Do not delete answer history when the respondent clears an answer.

Insert a new encrypted revision representing the cleared state.

### 10.4 Nonce generation

Generate a fresh 12-byte random nonce for every encrypted revision.

```python
nonce = secrets.token_bytes(12)
```

Never reuse a nonce with the same DEK.

The response database enforces:

```text
UNIQUE (envelope_id, nonce)
```

Because every historical nonce remains stored in the revision table, this constraint protects against accidental reuse across the full lifetime of the session DEK.

### 10.5 Additional authenticated data

Build stable Additional Authenticated Data, or AAD, for each encrypted revision.

Example conceptual content:

```text
crypto_version
envelope_id
answer_id
answer_locator
revision_id
revision_number
```

Example helper shape:

```python
def build_answer_aad(
    *,
    crypto_version: int,
    envelope_id: UUID,
    answer_id: UUID,
    answer_locator: bytes,
    revision_id: UUID,
    revision_number: int,
) -> bytes:
    ...
```

AAD is not encrypted, but its integrity is authenticated.

This prevents a database attacker from silently swapping encrypted revision rows.

### 10.6 Encryption operation

Conceptual Python:

```python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM

aesgcm = AESGCM(plaintext_dek)

ciphertext = aesgcm.encrypt(
    nonce,
    plaintext_payload_bytes,
    aad,
)
```

### 10.7 Decryption operation

```python
plaintext_payload_bytes = aesgcm.decrypt(
    revision.nonce,
    revision.ciphertext,
    aad,
)
```

If any of the following values are incorrect, decryption fails:

* key;
* nonce;
* ciphertext;
* authentication tag;
* AAD.

## 11. Plaintext-key caching

### 11.1 Why caching is useful

Requesting a plaintext DEK from KMS for every answer save would be wasteful.

A respondent may save many answers during one active session.

Cache plaintext DEKs temporarily in backend process memory.

### 11.2 Initial cache design

Use a bounded in-memory cache keyed by:

```text
response_envelope_id
```

Each cache entry contains:

```text
plaintext DEK
expiry time
```

Recommended behaviour:

```text
short TTL
bounded maximum size
least-recently-used eviction
no disk persistence
no database persistence
no logging
```

A process restart simply clears the cache. The backend can call KMS `Decrypt` again when needed.

### 11.3 Do not use a shared plaintext-key cache initially

Do not place plaintext DEKs in:

* PostgreSQL;
* Redis;
* a message queue;
* local files;
* environment variables;
* application logs.

A multi-instance deployment can allow each backend instance to unwrap and cache a DEK independently.

### 11.4 Linkage-secret cache

Use a separate in-memory cache for versioned linkage secrets retrieved from Secrets Manager.

AWS recommends client-side caching for Secrets Manager values because it improves speed and reduces API calls. [AWS-3]

The linkage-secret cache may have a longer TTL than the DEK cache because the secret is shared across sessions and changes only during intentional rotation.

### 11.5 Python memory limitation

Python cannot guarantee perfect cryptographic zeroisation of immutable byte objects.

The implementation should still minimise risk:

* avoid unnecessary copies;
* keep cache entries short-lived;
* remove references after use;
* never persist plaintext keys;
* never include them in exceptions or logs.
