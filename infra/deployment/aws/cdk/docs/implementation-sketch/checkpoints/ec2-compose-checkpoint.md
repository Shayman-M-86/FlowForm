## Compose Responsibilities

`compose.proxy.yml` runs only the proxy-side services on the public
proxy EC2:

- Caddy terminates public HTTP/HTTPS, obtains DNS-01 certificates with the
  Route 53 provider, and reverse proxies to `http://${APP_PRIVATE_IP}:5000`.
- Squid binds to `${PROXY_PRIVATE_IP}:3128` and allows CONNECT traffic only
  from `${SQUID_APP_SOURCE_CIDR}` to exact allow-listed domains.
- Caddy certificate state is persisted in Docker volumes; Squid logs are kept
  in a Docker volume.

`compose.app.yml` runs only the backend on the private app EC2:

- The backend image is supplied by `BACKEND_IMAGE`; production should pin it to
  an immutable digest.
- Docker publishes the backend only on `${APP_PRIVATE_IP}:5000:5000/tcp`;
  Gunicorn still listens on port `5000` inside the container.
- Backend outbound proxy env points at `http://${PROXY_PRIVATE_IP}:3128`.
- Secrets are file-mounted via the app-wide `*_FILE` convention.
- The container is read-only with tmpfs for `/tmp` and `/app/logs`, no Linux
  capabilities, `no-new-privileges`, a PID limit, and json-file log rotation.

The rehearsal stack (`infra/containers/strategies/rehearsal/compose/*.yml`, layered on
the shared runtime compose files) is the local/VM proof harness:

- Runs local Caddy, Squid, LocalStack, and a local registry instead of pulling
  from real AWS or ECR.
- Uses a separate PostgreSQL fixture VM with both databases; the app and DB
  stacks start independently, so readiness can recover once VM `240` is healthy.
- Substitutes a local TLS shim for certificate issuance.
- It does not prove AWS routing, IAM, endpoint policies, IMDS, RDS, or instance
  bootstrap.

## Expected EC2 Env Files

Proxy instance:

- `/opt/flowform/proxy.env`
- Expected to contain non-secret Compose interpolation values such as
  `CADDY_IMAGE`, `API_DOMAIN`, `AWS_REGION`, `PROXY_PRIVATE_IP`,
  `APP_PRIVATE_IP`, and `SQUID_APP_SOURCE_CIDR`.

App instance:

- `/opt/flowform/backend.env`
- Expected to contain non-secret Compose interpolation values and backend
  runtime config such as `BACKEND_IMAGE`, `APP_PRIVATE_IP`,
  `PROXY_PRIVATE_IP`, database host/name/user values, Auth0 IDs, KMS/linkage
  ARNs, SES config, and logging config. `PROXY_PRIVATE_IP` is used by Compose
  to derive backend `HTTP_PROXY`/`HTTPS_PROXY`; backend `NO_PROXY` is defined
  in `compose.app.yml`.

Env files should be generated by bootstrap, root-owned, not world-readable,
and free of secret values.

## Expected Secret Files

On the private app EC2, bootstrap should materialise these under
`/run/flowform/secrets` on tmpfs:

- `DATABASE_CORE_APP_PASSWORD.secret.txt`
- `DATABASE_RESPONSE_APP_PASSWORD.secret.txt`
- `FLOWFORM_APP_SECRET_KEY.secret.txt`
- `FLOWFORM_AUTH0_MGMT_SECRET.secret.txt`

Compose maps those files into the backend container as Docker secrets at
`/run/secrets/...`, and the backend reads them through:

- `DATABASE_CORE_APP_PASSWORD_FILE`
- `DATABASE_RESPONSE_APP_PASSWORD_FILE`
- `FLOWFORM_APP_SECRET_KEY_FILE`
- `FLOWFORM_AUTH0_MGMT_SECRET_FILE`

The linkage secret is different: the backend receives the linkage secret ARN
through non-secret config and fetches the versioned secret at runtime with its
instance role.

## Not Proven Yet

Still needs implementation or staging evidence:

- CDK VPC/subnet route tables for public proxy and private app instances.
- Public proxy EC2 with Elastic IP.
- Private app EC2 with no public IP and no broad internet route.
- S3 gateway endpoint and intentionally scoped endpoint policy for ECR layers.
- Proxy, app, RDS, and EICE security groups.
- Separate proxy/app instance roles and least-privilege policies.
- IMDSv2 required with hop limit 2 for both instances.
- Host bootstrap writing `/opt/flowform/proxy.env`,
  `/opt/flowform/backend.env`, and tmpfs secret files.
- Docker daemon proxy configuration on the private app instance.
- Host firewalls mirroring security groups.
- Real ECR image pulls after lockdown.
- Caddy Route 53 DNS-01 issuance and renewal in staging.
- Backend AWS calls through Squid and direct private/VPC paths for IMDS, RDS,
  DNS/NTP, and S3 gateway endpoint access.
- Public API health through Caddy and private backend port isolation.

The next implementation pass should start the CDK network/security skeleton,
not backend runtime code:

- Update `network_stack.py` away from the stale ECS/ALB/NAT comments and toward
  the two-instance model.
- Add the public proxy subnet/instance path with Elastic IP.
- Add the private app subnet/instance path with no public IP and no NAT route.
- Add the S3 gateway endpoint for ECR layer downloads.
- Define proxy, app, RDS, and EICE security group relationships:
  - proxy inbound public `80`/`443`;
  - proxy inbound private `3128` only from the app SG;
  - app inbound `5000` only from the proxy SG;
  - app outbound `3128` to proxy, `5432` to RDS, and S3 endpoint path;
  - RDS inbound `5432` only from the app SG.
- Require IMDSv2 with hop limit 2.
- Keep proxy and app instance roles separate.

## Do Not Change Casually

- No broad app internet egress.
- No NAT shortcut for the app subnet.
- No broad Squid wildcards such as `.amazonaws.com` or `.auth0.com`.
- No raw environment-variable secrets; keep the file-backed `*_FILE` pattern.
- No production builds on EC2; CI builds images, EC2 pulls immutable refs.
- No public backend port binding.
- No ALB, PgBouncer, Redis, or workers until a real need triggers them.
